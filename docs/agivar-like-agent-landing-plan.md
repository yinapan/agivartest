# 类 Agivar 桌面版 Agent 落地方案

## 1. 背景与目标

目标产品参考 Agivar 的公开文档与产品形态，核心能力是让用户通过对话、文字教学和录屏演示，把重复的桌面或浏览器业务流程教给 Agent。Agent 在后续任务中检索这些流程记忆，并自动操作电脑完成任务。

本方案按你的最新方向调整为：**直接做桌面版**。这意味着第一版就要具备桌面窗口感知、屏幕观察、鼠标键盘控制、录屏教学、人工接管和本地记忆库能力。

为了避免范围失控，桌面版 MVP 仍然需要有清晰边界：

- 第一版优先支持 Windows。
- 第一版优先支持 2-3 个明确应用或业务流程。
- 浏览器流程作为桌面 Agent 的一个内置能力，而不是单独产品方向。
- 全局任意软件泛化能力后置，但底层架构从第一天为桌面泛化预留。

第一阶段要验证的不是“任何软件都能自动操作”，而是：

- 用户能通过文字和录屏把一个桌面流程教给系统。
- 系统能把教学内容沉淀为可检索、可编辑、可执行的流程记忆。
- 系统能观察屏幕、定位控件、执行点击输入、验证状态。
- 遇到登录、验证码、权限、支付、删除、提交等风险动作时，系统能暂停并交给用户。

## 2. 产品定位

### 2.1 一句话定位

面向企业重复电脑流程的可教学桌面 Agent：用户演示一次、补充规则，系统以后能按记忆半自动操作电脑完成同类任务。

### 2.2 目标用户

- 运营人员：电商后台、内容平台、数据下载、批量处理。
- 财务人员：发票录入、报销提交、对账、银行流水整理。
- 行政和人事：系统填报、员工信息维护、表格转录。
- 销售和客服：CRM、工单系统、客户资料维护。
- 内部流程优化团队：把高频人工流程沉淀为可复用自动化。

### 2.3 首选试点场景

桌面版第一阶段要选“真实但可控”的流程：

- Windows 上的浏览器后台操作。
- Excel / CSV 数据读取，加网页或桌面系统录入。
- 企业微信、钉钉、飞书等客户端中的固定流程。
- 内部 Windows 客户端软件中的查询、录入、导出。
- 文件夹、下载目录、Excel、浏览器之间的跨应用流转。

第一阶段不建议选择：

- 银行转账、支付、批量删除、批量发消息等高风险动作。
- 强依赖验证码、人脸、USB Key、短信 2FA 的流程。
- 图像内容变化极大且没有稳定文字或控件结构的流程。
- 游戏、远程桌面、虚拟机内复杂操作。

## 3. Agivar 能力拆解

| 模块 | Agivar 公开表现 | 桌面版实现启发 |
| --- | --- | --- |
| 桌面客户端 | Windows/macOS 客户端、聊天区、任务进度、录屏浮窗 | 需要一个常驻桌面壳层和任务控制面板 |
| 任务模式 | 用户输入目标，Agent 规划、执行、观察、修正 | 需要 Planner、Executor、Observer 闭环 |
| 教学模式 | 用户讲规则、偏好、流程，系统整理为长期记忆 | 需要结构化记忆库和可编辑流程 |
| 录屏教学 | 采集屏幕、鼠标、键盘、注释，生成流程 | 需要录屏、事件采集、关键帧抽取、多模态理解 |
| 记忆库 | 按平台和主题组织流程、规则和偏好 | 需要本地存储、向量检索、版本管理 |
| GUI 执行 | 操作网站或桌面软件，必要时让用户接管 | 需要 UI Automation、视觉定位、输入控制和人工暂停 |

关键点：录屏教学不应沉淀为坐标回放，而应沉淀为“语义流程”。每一步都要有目标、定位依据、成功判断和失败处理。

## 4. 桌面版 MVP 范围

### 4.1 必须包含

1. **桌面客户端**

   使用 Electron 或 Tauri 构建 Windows 桌面应用，提供任务模式、教学模式、录屏教学、记忆库管理、任务日志和人工接管浮窗。

2. **屏幕观察能力**

   支持当前屏幕截图、活动窗口识别、窗口标题识别、窗口区域截图。第一版可以优先支持主屏幕，不强求多显示器完善体验。

3. **Windows UI Automation 能力**

   通过 UI Automation 读取窗口树、控件名称、控件类型、可点击区域、文本框、按钮、菜单等信息。能用控件树定位时优先使用控件树。

4. **视觉理解能力**

   当 UI Automation 不可用或信息不足时，用截图 + OCR + 多模态模型识别目标区域，例如按钮、菜单、表格、图标附近文字。

5. **鼠标键盘控制**

   支持点击、双击、拖拽、输入、快捷键、复制粘贴、滚动。所有输入动作要写入任务日志。

6. **浏览器执行适配**

   浏览器流程仍然很重要。桌面版中内置 Playwright 或 CDP 适配器，用于稳定控制浏览器页面。浏览器不是产品边界，而是桌面 Agent 的一个高稳定执行通道。

7. **文字教学**

   用户可以用自然语言教流程、规则、字段含义和注意事项。系统把这些内容结构化为流程记忆。

8. **录屏教学**

   支持录制屏幕区域或活动窗口，同步采集鼠标点击、键盘输入摘要、窗口变化和用户文字注释。MVP 可以先不做语音注释。

9. **流程记忆库**

   使用 SQLite 和本地文件目录存储流程、录屏、关键帧、执行日志和版本。流程必须可查看、可编辑、可禁用、可回滚。

10. **人工接管**

   登录、验证码、支付、删除、提交审批、批量发送等风险节点强制暂停。用户处理后点击继续，系统重新观察状态再恢复执行。

### 4.2 第一版不做

- 不承诺能操作所有桌面软件。
- 不做无人值守高风险操作。
- 不自动处理验证码、短信、人脸、USB Key。
- 不做 macOS 全能力支持。
- 不做团队协作、云同步、权限审计。
- 不做复杂脚本市场或插件生态。

## 5. 桌面 Agent 技术架构

```text
桌面客户端
  -> 聊天与任务面板
  -> 教学模式
  -> 录屏浮窗
  -> 记忆库编辑器
  -> 人工接管浮窗

本地 Agent 服务
  -> Planner：任务规划
  -> Memory Retriever：记忆检索
  -> Desktop Observer：屏幕与窗口观察
  -> Desktop Executor：鼠标键盘执行
  -> UIA Adapter：Windows UI Automation
  -> Browser Adapter：Playwright / CDP
  -> Recorder：录屏与事件采集
  -> Workflow Extractor：录屏转流程
  -> Safety Guard：风险动作识别与暂停

存储层
  -> SQLite：会话、任务、记忆、步骤、版本
  -> 本地文件目录：录屏、关键帧、截图、日志
  -> 向量索引：流程记忆检索

模型层
  -> 文本模型：规划、总结、流程结构化
  -> 视觉模型：截图理解、状态判断
  -> Embedding 模型：记忆检索
```

## 6. 推荐技术选型

| 层级 | MVP 推荐 | 说明 |
| --- | --- | --- |
| 桌面壳 | Electron + React + TypeScript | Windows 桌面集成和开发效率更稳 |
| 本地服务 | Node.js + TypeScript | 与 Electron、Playwright、任务日志集成顺畅 |
| Windows UI Automation | Python `uiautomation` / pywinauto，或 .NET UIA Bridge | 可先用独立本地 worker，再封装 RPC |
| 浏览器控制 | Playwright | 浏览器流程稳定性远高于纯视觉点击 |
| 屏幕截图 | Electron desktopCapturer / Windows Graphics Capture | 先满足主屏和活动窗口截图 |
| 输入控制 | robotjs、nut.js、AutoHotkey Bridge 或 Windows SendInput | MVP 可先选最稳定的一条 |
| OCR | PaddleOCR、Tesseract 或云端视觉模型 | 中文场景建议验证 PaddleOCR |
| 存储 | SQLite + 本地文件目录 | 单机桌面产品足够 |
| 向量索引 | sqlite-vec / LanceDB / Qdrant local | 先本地化，降低隐私顾虑 |
| 录屏 | Windows Graphics Capture / FFmpeg | MVP 优先可用，后续优化性能 |

工程上建议把桌面能力封装为统一工具接口：

```text
observe_screen()
observe_active_window()
get_ui_tree(window_id)
click(target)
type_text(text)
press_hotkey(keys)
scroll(direction, amount)
wait_for(condition)
ask_user_takeover(reason)
```

上层 Agent 不直接关心底层是 UIA、Playwright、OCR 还是坐标点击。

## 8. 执行策略

### 8.1 定位优先级

桌面 Agent 不应一上来就用截图点坐标。建议按以下优先级执行：

1. **浏览器 DOM 定位**

   如果目标在浏览器中，优先使用 Playwright 的 role、label、text、placeholder、selector。

2. **Windows UI Automation 定位**

   如果目标是原生或类原生桌面应用，优先通过控件树定位按钮、文本框、菜单、列表。

3. **OCR + 视觉定位**

   如果控件树不可用，用截图、OCR 和视觉模型定位目标区域。

4. **短期坐标兜底**

   坐标只能作为当前会话兜底，不作为长期流程记忆主路径。

5. **人工接管**

   如果系统信心不足或动作风险高，暂停并让用户处理。

### 8.2 每步必须验证

每个步骤都要有 `expected_state`：

- 出现某个窗口标题。
- 页面出现某段文字。
- 表格新增一行。
- 按钮变为禁用或状态变成已提交。
- 文件下载到指定目录。
- 输入框内容与预期一致。

没有验证条件的步骤，不允许作为稳定自动化步骤。

### 8.3 风险动作分级

| 风险等级 | 示例 | 策略 |
| --- | --- | --- |
| 低 | 查询、筛选、复制、导出、打开文件 | 可自动执行 |
| 中 | 新增记录、修改字段、上传附件 | 执行前展示计划，必要时确认 |
| 高 | 删除、提交审批、批量发送、覆盖文件 | 强制人工确认 |
| 禁止 | 代输验证码、绕过 2FA、自动支付、自动转账 | 必须人工接管 |

## 9. 录屏教学流程

1. 用户选择教学模式。
2. 用户选择应用或窗口，例如“Excel + 某 CRM 客户端”。
3. 用户点击开始录屏。
4. 系统记录：
   - 屏幕视频。
   - 活动窗口标题。
   - 鼠标点击、拖拽、滚动。
   - 键盘输入摘要，不记录密码明文。
   - 用户文字注释。
   - 关键帧截图。
5. 系统生成流程草稿：
   - 流程目标。
   - 初始状态。
   - 步骤列表。
   - 每步定位策略。
   - 成功判断。
   - 风险动作。
   - 失败处理建议。
6. 用户在流程编辑器中确认或修改。
7. 系统写入记忆库。

## 10. 安全与隐私

桌面版会天然接触更多敏感信息，因此安全能力必须从 MVP 开始做。

必须包含：

- 默认本地存储，不默认上传完整录屏。
- 模型调用前只上传必要截图和文本，并做脱敏。
- 密码框、验证码、支付信息、身份证号、银行卡号默认遮挡或跳过。
- 高风险动作强制暂停。
- 任务日志可删除。
- 录屏前明确提示采集范围。
- 用户可选择仅录活动窗口，避免录到其他应用。
- 所有自动输入和点击都有可追踪日志。

## 11. 交付排期

### 阶段 0：桌面能力验证，1-2 周

目标：确认 Windows 桌面控制链路可行。

交付物：

- 截取主屏和活动窗口。
- 识别活动窗口标题。
- 读取基础 UIA 控件树。
- 执行点击、输入、快捷键。
- 对浏览器走 Playwright，对桌面应用走 UIA 或视觉。

验收标准：

- 能自动打开记事本、输入文本、保存文件。
- 能识别并点击一个 Windows 原生按钮。
- 能在浏览器中打开页面并完成一次表单填写。

### 阶段 1：桌面任务闭环，2-3 周

目标：跑通“目标 -> 计划 -> 观察 -> 执行 -> 验证 -> 日志”闭环。

交付物：

- Electron 桌面客户端。
- 任务输入框。
- 本地 Agent 服务。
- 屏幕观察和执行工具。
- 任务步骤日志。
- 人工暂停与继续。

验收标准：

- 2 个固定桌面流程可重复执行。
- 失败时能定位到步骤和原因。
- 用户可以手动接管后继续。

### 阶段 2：文字教学与流程记忆，2-3 周

目标：用户能教流程，系统能沉淀并复用。

交付物：

- 教学模式。
- 结构化流程记忆。
- 记忆检索。
- 流程编辑器。
- 版本回滚。

验收标准：

- 用户无需改代码即可修改步骤。
- 任务前能命中正确流程记忆。
- 同一流程可稳定复用。

### 阶段 3：录屏教学，4-5 周

目标：把用户示范转成可编辑流程草稿。

交付物：

- 活动窗口录屏。
- 鼠标键盘事件采集。
- 关键帧抽取。
- 流程草稿生成。
- 用户审核入库。

验收标准：

- 5 条录屏中至少 3 条能生成可编辑流程。
- 用户编辑后可执行。
- 录屏、关键帧、流程步骤能关联查看。

### 阶段 4：试点打磨，4 周

目标：在真实业务流程中验证价值。

交付物：

- 3-5 条真实桌面流程模板。
- 批量输入能力。
- 失败重试。
- 执行报告。
- 安全暂停策略优化。

验收标准：

- 核心流程连续执行 20 次，成功率达到 80%以上。
- 高风险动作暂停准确率达到 100%。
- 每次失败都有明确步骤和原因。
- 单条流程平均节省人工时间超过 50%。

## 12. MVP 成功指标

### 产品指标

- 用户完成一条流程教学所需时间。
- 流程第 2 次复用成功率。
- 每 100 步操作中的人工接管次数。
- 单条流程平均节省时间。
- 用户愿意长期维护的流程数量。

### 工程指标

- UIA 控件定位成功率。
- 浏览器 DOM 定位成功率。
- 视觉定位成功率。
- 步骤级验证准确率。
- 失败可诊断率。
- 单任务模型调用成本。

### 安全指标

- 高风险动作暂停准确率。
- 敏感输入落盘次数，目标为 0。
- 录屏采集范围误录次数。
- 用户删除日志和录屏是否即时生效。

## 13. 主要风险与应对

| 风险 | 表现 | 应对 |
| --- | --- | --- |
| 桌面泛化过难 | 不同软件控件结构差异大 | 第一版限定 Windows + 2-3 个应用场景 |
| 纯视觉点击不稳定 | 页面或窗口位置变化导致失败 | UIA / DOM 优先，视觉兜底，坐标只短期使用 |
| 权限复杂 | 截屏、输入控制、录屏被系统拦截 | 阶段 0 先验证权限链路 |
| 安全顾虑 | 用户担心录屏和输入被记录 | 本地优先、脱敏、活动窗口录制、日志可删除 |
| 录屏理解不准 | 生成步骤遗漏或误解 | 强制流程编辑器和用户审核 |
| 执行不可控 | 误提交、误删除、误发消息 | 风险分级和强制人工确认 |
| 模型成本高 | 每一步都调用视觉模型太贵 | 稳定流程编译为动作计划，模型主要处理异常 |

## 14. 更稳的桌面版 MVP 定义

先做一个 **Windows 桌面可教学 Agent**：

- 一个 Electron 桌面客户端。
- 一个本地 Agent 服务。
- 支持屏幕观察、活动窗口识别、UIA 控件读取。
- 支持鼠标键盘控制。
- 支持浏览器 Playwright 适配。
- 支持文字教学和流程编辑器。
- 支持活动窗口录屏教学。
- 支持本地流程记忆库。
- 支持人工接管。
- 支持 3-5 条真实流程模板。

明确不交付：

- 不交付 macOS 完整支持。
- 不交付任意软件泛化承诺。
- 不交付无人值守高风险操作。
- 不交付验证码和支付自动化。
- 不交付团队协作和云端管理。

## 15. 下一步行动

1. 选定 Windows 作为第一平台。
2. 选定 2-3 个试点应用，例如浏览器后台、Excel、某个企业客户端。
3. 用阶段 0 验证截屏、UIA、输入控制、Playwright 四条链路。
4. 建立 10 条真实桌面流程评测集。
5. 先实现手写流程记忆执行，再实现文字教学，最后接入录屏教学。
6. 从第一天记录执行日志、截图证据、风险暂停和失败原因。
