# Claude 落地方案改进建议

## 1. 评审对象

- 文件：[落地方案.md](../落地方案.md)
- 主题：Agivar 桌面 AI 助手克隆落地方案
- 评审目的：判断该方案是否适合作为第一版产品的执行依据，并给出范围收缩、技术路径和商业模型上的改进建议。

## 2. 总体判断

Claude 的方案整体质量较高，已经覆盖产品分析、竞品参考、技术架构、录屏引擎、Computer Use 执行、记忆库、团队配置、成本估算、风险和成功指标。

它的问题不在于“不完整”，而在于**第一版过于完整**。如果按原方案直接执行，早期团队会同时面对桌面自动化、录屏处理、多模态理解、云端同步、用户系统、脚本沙箱、成本控制和商业化定价等多个高风险模块，容易出现每个模块都有，但端到端体验不稳定的情况。

建议把该方案从“6 个月完整产品路线”拆成“先验证桌面执行闭环，再逐步叠加教学和录屏能力”的路线。

## 3. 必须修改

### 3.1 不要使用“克隆”作为产品表述

原文标题是“Agivar 桌面 AI 助手克隆”。这个表述不建议继续使用。

原因：

- 容易引发法务和品牌风险。
- 对外不利于建立独立定位。
- 对内也容易让团队以“复刻功能列表”为目标，而不是围绕目标用户的问题做取舍。

建议改为：

- 类 Agivar 桌面 Agent 落地方案
- 可教学桌面流程 Agent 落地方案
- 企业桌面流程自动化 Agent 产品方案

### 3.2 收缩 P0 范围

原方案阶段 1 的 P0 同时包含：

- Electron 桌面客户端
- 录屏引擎
- 录制浮窗
- 文字注释
- 录屏处理
- 录屏预览
- 记忆库
- 教学模式
- 用户认证
- 后端基础设施

这个 P0 太大。第一阶段应该优先证明“系统能稳定操作桌面并可恢复”，而不是同时做完整教学产品。

建议 P0 只保留：

- 桌面壳
- 截图与活动窗口识别
- Windows UI Automation 控件读取
- Playwright 浏览器适配
- 鼠标键盘执行
- 手写流程记忆
- 任务日志
- 人工暂停与继续
- 紧急停止

录屏教学、账号系统、云同步、OSS、ASR、视频问答后置。

### 3.3 调整阶段顺序

原方案采用“先教学侧 MVP，再执行侧 MVP”的顺序。这个顺序有明显风险：如果执行闭环不稳定，录屏教学生成的流程再漂亮，也不能转化为真实价值。

建议改为：

1. 桌面控制 PoC
2. 手写流程记忆执行
3. 文字教学生成流程草稿
4. 流程编辑器
5. 录屏教学生成流程草稿
6. 云同步和团队能力

核心原则：先证明能执行，再降低教学成本。

### 3.4 云端架构不应进入第一版主路径

原方案很早引入：

- API Gateway
- 用户认证
- OSS / S3
- PostgreSQL + pgvector
- Redis + Asynq
- ASR 服务
- 视频问答服务
- 云端记忆库同步

这些能力适合企业版和规模化阶段，但不是桌面 Agent 第一版的最短路径。

建议第一版本地优先：

- SQLite 存流程、任务和记忆。
- 本地文件目录存录屏、截图和日志。
- 模型调用只上传必要截图和文本。
- 登录、云同步、团队空间后置。

### 3.5 成本与定价模型不匹配

原方案估算降本后单用户成本约 ¥120-180/月，但个人版定价为 ¥99/月，同时还提供免费版每日任务和录屏额度。

这里存在商业模型冲突。

建议改为额度制：

- 按任务次数计费。
- 按录屏处理分钟数计费。
- 按多模态模型调用次数计费。
- 免费版只提供本地流程编辑和少量演示额度。

如果保留订阅制，也要设置明确的模型调用上限，避免高频用户导致亏损。

### 3.6 Computer Use 成功率目标需要拆分

原方案中“Computer Use 执行”用简单任务和中等任务成功率衡量，但这个指标过粗。

桌面执行至少要拆成：

- Playwright 浏览器 DOM 定位成功率
- Windows UI Automation 控件定位成功率
- OCR + 视觉定位成功率
- 坐标兜底成功率
- 人工接管触发准确率
- 步骤级验证准确率

否则无法知道成功率提升来自哪里，也无法定位失败原因。

### 3.7 Python 脚本沙箱不要作为第一版能力

原方案设计了 Python 脚本沙箱，包括 `RestrictedPython`、import 白名单和文件系统限制。

这个方向有价值，但不适合作为第一版核心能力。桌面 Agent 已经具备较高权限，再允许模型生成脚本执行，会显著增加安全风险。

建议第一版只允许固定工具调用：

- 文件读取
- CSV 读取
- 文件复制
- 浏览器操作
- UIA 控件操作
- 鼠标键盘动作

脚本执行后置，并且必须满足：

- 用户显式批准。
- 运行前展示代码。
- 限制工作目录。
- 默认无网络。
- 默认无系统命令执行。

### 3.8 紧急停止热键需要更换

原方案默认紧急停止热键为 `Ctrl+Shift+Esc`，这是 Windows 打开任务管理器的系统快捷键。

建议改为：

- `Ctrl+Alt+Space`
- 或允许用户自定义。

同时保留：

- 托盘菜单停止按钮。
- 悬浮窗停止按钮。
- 执行前显示停止方式。

## 4. 建议补充

### 4.1 增加目标应用矩阵

需要明确第一版支持哪些应用和执行策略。

建议表格：

| 应用 | 场景 | 首选定位方式 | 兜底方式 |
| --- | --- | --- | --- |
| Chrome / Edge | 后台网页流程 | Playwright | OCR + 视觉 |
| Excel | 读取表格、复制数据 | COM / UIA | 文件解析 |
| 文件资源管理器 | 文件选择、下载目录处理 | UIA | 坐标兜底 |
| 企业微信 / 钉钉 / 飞书 | 固定客户端流程 | UIA + OCR | 人工接管 |
| 内部 Windows 客户端 | 查询、录入、导出 | UIA | 视觉定位 |

### 4.2 增加固定评测集

必须建立 10-20 条真实桌面流程作为评测集，用于每次回归。

每条评测流程应记录：

- 初始状态。
- 操作目标。
- 允许使用的工具。
- 成功条件。
- 高风险动作。
- 失败时应如何暂停。
- 期望耗时。

没有评测集，模型和 Prompt 每次调整都只能靠主观演示判断。

### 4.3 把流程编辑器提升为核心模块

录屏和文字教学生成的流程一定会有错。流程编辑器不是体验优化项，而是 MVP 必需项。

流程编辑器至少要支持：

- 查看步骤。
- 修改步骤意图。
- 修改定位策略。
- 修改输入来源。
- 修改成功判断。
- 标记风险等级。
- 禁用某一步。
- 回滚历史版本。

### 4.4 录屏技术路线需要重新评估

原方案 Windows 录屏首选 DXGI Desktop Duplication。建议同时评估 Windows Graphics Capture。

理由：

- WGC 对现代 Windows 版本和窗口级捕获更友好。
- 更适合“只录目标窗口”的隐私诉求。
- DXGI 在多屏、HDR、DPI、窗口排除方面工程复杂度较高。

建议阶段 0 同时验证 DXGI 和 WGC，最后按稳定性、权限、窗口级捕获和性能选择。

### 4.5 Undo 机制需要降级表述

原方案提出 Undo 栈。这个概念容易让用户误以为所有操作都能撤销。

真实情况：

- 文件操作可以做快照。
- 网页提交通常无法撤销。
- 第三方系统写入无法保证回滚。
- 发送消息、提交审批、支付类动作无法撤销。

建议改成：

- 文件操作快照。
- 执行前确认。
- 高风险动作暂停。
- 可回放日志。
- 失败重试。

不要承诺通用 Undo。

### 4.6 增加权限和隐私引导

桌面 Agent 第一次启动时必须解释清楚：

- 需要哪些权限。
- 为什么需要。
- 截图和录屏范围是什么。
- 哪些数据会本地保存。
- 哪些数据会发送给模型。
- 用户如何删除数据。
- 如何随时停止 Agent。

这是用户信任的关键，不是合规文档里的附属内容。

## 5. 可以后置

以下能力可以保留在路线图，但不应进入第一版：

- 语音注释。
- 视频问答。
- 模板市场。
- 团队记忆库。
- macOS / Linux 完整支持。
- 本地多模态大模型。
- 自动更新。
- 云端同步。
- ASR 服务。
- 复杂 Python 脚本沙箱。
- 企业管理后台。
- 插件生态。

## 6. 建议重排后的 MVP

### 6.1 MVP 定义

第一版做一个 Windows 桌面可教学 Agent 原型。

核心能力：

- Electron 桌面壳。
- 截屏与活动窗口识别。
- Windows UI Automation 控件读取。
- Playwright 浏览器适配。
- 鼠标键盘执行。
- 手写流程记忆。
- 文字教学生成流程草稿。
- 流程编辑器。
- 执行日志和截图证据。
- 人工接管和紧急停止。
- 3-5 条真实流程模板。

### 6.2 MVP 不包含

- 不包含任意软件泛化承诺。
- 不包含云端同步。
- 不包含团队空间。
- 不包含录屏教学作为核心验收项。
- 不包含语音注释。
- 不包含自动执行高风险动作。
- 不包含 Python 任意脚本执行。

### 6.3 建议里程碑

| 阶段 | 周期 | 目标 | 验收 |
| --- | --- | --- | --- |
| 阶段 0 | 1-2 周 | 桌面控制 PoC | 可截图、读 UIA、点击输入、控制浏览器 |
| 阶段 1 | 2-3 周 | 手写流程执行 | 2-3 条固定流程可重复执行 |
| 阶段 2 | 2-3 周 | 文字教学和流程编辑器 | 用户能改流程并重新运行 |
| 阶段 3 | 3-4 周 | 轻量录屏教学 | 录屏可生成流程草稿，但必须人工审核 |
| 阶段 4 | 4 周 | 真实试点打磨 | 3-5 条真实流程成功率达到 80%以上 |

## 7. 推荐落地顺序

1. 先选 Windows 作为唯一第一平台。
2. 选定 2-3 个目标应用，例如 Chrome / Edge、Excel、某个企业客户端。
3. 建立 10 条真实桌面流程评测集。
4. 先实现手写流程执行，不先做录屏。
5. 让流程编辑器成为记忆生成和执行之间的必经环节。
6. 再接入文字教学。
7. 最后接入录屏教学。
8. 云端同步、账号、ASR、视频问答全部后置。

## 8. 结论

Claude 的方案适合作为完整产品路线图，但不适合作为第一版执行清单。

建议保留它的架构思想、定位优先级、风险动作分级、数据模型和成本意识；同时收缩第一版范围，把执行闭环放在录屏教学之前，把本地优先放在云端同步之前，把流程编辑器放在核心链路中。

最关键的改动是：

> 第一版不要证明“AI 能看录屏学会一切”，而要证明“用户给出一个明确桌面流程后，系统能稳定执行、验证、暂停和复用”。
