# 新版落地方案二次改进建议

## 1. 评审对象

- 文件：[落地方案.md](../落地方案.md)
- 当前标题：可教学桌面流程 Agent — 落地方案
- 评审目的：基于已经修改过的新版方案，继续检查 MVP 边界、技术路线、商业模型和风险表述是否足够收口。

## 2. 总体判断

新版方案已经比上一版更可执行：

- 标题去掉了“克隆”表述。
- 阶段顺序改成“先执行闭环，再教学，再录屏”。
- 增加了目标应用矩阵。
- 增加了固定评测集要求。
- Python 脚本执行已经后置。
- 增加了首次启动权限和隐私引导。
- 紧急停止热键已改为 `Ctrl+Alt+Space`。

这些调整方向正确。现在的主要问题不是大方向，而是**文档前后仍有一些商业版能力混在 MVP 主路径里**。如果不继续收口，研发团队仍可能把账号、云同步、OSS、ASR、视频问答、自动更新等能力提前做进去，导致第一版交付变慢。

## 3. 必须修改

### 3.1 统一“本地优先”和“需要联网”的表述

当前 `1.7 系统要求` 写到：

- 需要联网（登录、云端同步、录屏 LLM 处理）
- 离线模式下可使用本地记忆库执行已学流程

这与后文“本地优先、云同步后置”的路线存在冲突。

建议改为：

- 第一版核心功能本地可用。
- 联网只用于模型调用和可选更新。
- 登录、云端同步、跨设备记忆库不是 MVP 必需能力。
- 已沉淀流程应支持离线执行。

### 3.2 拆分 MVP 架构图和商业版架构图

当前 `3.1 整体架构图` 仍然把以下内容放在主架构中：

- 云端服务层
- 用户认证
- ASR
- 视频问答
- 录屏处理流水线
- Python 沙箱

这些不应出现在 MVP 主路径图里。

建议拆成两张图：

1. **MVP 架构图**
   - Electron 桌面客户端
   - 本地 Agent 服务
   - 截屏 / UIA / Playwright / 输入控制
   - SQLite + 本地文件目录
   - 模型 API 调用适配层
   - 安全暂停和执行日志

2. **后续商业版架构图**
   - 用户认证
   - 云同步
   - OSS / S3
   - ASR
   - 视频问答
   - 团队记忆库
   - 企业审计

这样团队不会把完整商业架构误认为第一版交付范围。

### 3.3 阶段 3 录屏教学仍然偏重

新版已把录屏教学放到阶段 3，这是正确的。但阶段 3 的 P0 仍然包含：

- API Gateway
- 录屏处理流水线
- OSS
- 后端基础

建议阶段 3 先做轻量版：

- 本地录屏。
- 本地关键帧抽取。
- 本地事件时间线。
- 直接调用多模态模型生成流程草稿。
- 用户在流程编辑器中审核后入库。

云端流水线、OSS、账号绑定后置到阶段 4 或阶段 5。

### 3.4 第一版目标应用矩阵需要去掉高风险场景

当前第一版目标应用矩阵包括企业微信、钉钉、飞书，场景写的是“消息、审批”。

这不适合作为第一版目标场景：

- 消息发送容易造成误发。
- 审批提交属于高风险业务动作。
- 企业 IM 客户端 UI 变化频繁，自动化稳定性不一定好。
- 用户信任成本高。

建议第一版目标应用调整为：

| 应用 | 是否保留 | 理由 |
| --- | --- | --- |
| Chrome / Edge | 保留 | Playwright 稳定，适合跑通闭环 |
| Excel | 保留 | 企业高频数据源 |
| 文件资源管理器 | 保留 | 文件流转是桌面 Agent 基础能力 |
| 记事本 / 写字板 | 保留 | 桌面控制 PoC 简单稳定 |
| 内部低风险 Windows 客户端 | 保留 | 可作为真实试点 |
| 企业微信 / 钉钉 / 飞书 | 后置 | 消息和审批动作风险高 |

如果必须做 IM 客户端，第一版只做“读取、搜索、打开指定页面”，不要做发送和审批。

### 3.5 成本模型仍然不健康

当前文档写：

- 降本后成本约 ¥120-180/用户/月。
- 个人版 ¥99/月。
- 每天 20 次任务。
- 超额 ¥3/次。
- 依靠大部分用户日均使用量低于 5 次来盈利。

这个模型风险仍然偏高。桌面 Agent 的高频用户很可能正是最有付费意愿的用户，也最容易把成本打穿。

建议改为：

- 订阅只覆盖基础功能和本地执行。
- 云端多模态调用单独消耗任务点数。
- 录屏 AI 解析按分钟或关键帧额度计费。
- 专业版不要写“无限录屏”，改成“本地录屏不限，AI 解析按额度”。
- 首版定价宁可偏保守，不要用高频亏损换增长。

### 3.6 成功指标要按阶段拆分

当前 `11.1 产品指标` 仍写：

- 用户完成一条流程教学所需时间 < 15 分钟，包括录屏 + 编辑确认

但新版路线中录屏教学是阶段 3，不是阶段 1 和阶段 2 的核心。

建议拆成：

| 阶段 | 指标 |
| --- | --- |
| 阶段 1 | 手写流程从导入到执行成功 < 10 分钟 |
| 阶段 2 | 文字教学生成流程草稿 < 5 分钟 |
| 阶段 2 | 用户修改流程后可重新运行 |
| 阶段 3 | 录屏结束到流程草稿生成 < 3 分钟 |
| 阶段 3 | 5 条录屏中至少 3 条生成可执行草稿 |

这样验收标准和阶段目标更一致。

### 3.7 风险表需要区分 MVP 风险和后续风险

当前风险表仍混在一起，包括：

- macOS 权限
- 跨平台一致性
- Python 脚本安全
- 云端隐私合规

这些对第一版 Windows MVP 不是核心风险。建议拆成：

1. **MVP 核心风险**
   - UIA 控件读取不稳定。
   - 截图和 DPI 映射错误。
   - Playwright 和用户真实浏览器会话打通困难。
   - 高风险动作误执行。
   - 任务失败无法诊断。
   - 用户不信任桌面权限。

2. **后续阶段风险**
   - macOS 权限。
   - 云同步合规。
   - Python 脚本安全。
   - 团队权限审计。
   - 本地大模型性能。

## 4. 建议补充

### 4.1 增加“第一版不做什么”

建议在第五章 MVP 功能规划前增加明确边界：

- 不做登录和云同步。
- 不做团队空间。
- 不做视频问答。
- 不做语音注释。
- 不做 Python 脚本执行。
- 不做 macOS。
- 不做消息群发。
- 不做审批提交。
- 不做支付、转账、删除等高风险自动化。
- 不承诺任意软件可操作。

这能防止试点和研发过程中不断加范围。

### 4.2 增加“最小可跑 Demo”定义

建议定义一个 2 周内必须完成的 Demo：

> 用户在桌面客户端输入目标，系统读取一条手写流程，在 Windows 上完成“打开浏览器 / 打开网页 / 填写表单 / 保存执行日志”的闭环。失败时可以暂停，用户接管后继续。

这个 Demo 应该成为阶段 0 和阶段 1 的交界验收物。

### 4.3 增加真实浏览器会话方案

桌面 Agent 做浏览器任务时，是否使用用户已有浏览器会话很关键。

需要明确：

- 使用 Playwright 自带 Chromium，还是连接用户 Chrome / Edge？
- 如果要复用登录态，如何处理用户数据目录？
- 如何避免污染用户日常浏览器？
- 如何处理插件、企业 SSO、证书和代理？

建议第一版支持两种模式：

- 默认使用独立托管浏览器上下文。
- 需要登录态时，由用户在托管浏览器里登录。

不要直接控制用户主浏览器，除非有明确授权和隔离策略。

### 4.4 增加“状态验证 DSL”

文档多次强调 `expected_state`，但还缺少具体表达方式。

建议定义最小状态验证 DSL：

```yaml
expected_state:
  any:
    - type: window_title_contains
      value: "客户管理"
    - type: page_text_contains
      value: "保存成功"
    - type: file_exists
      path: "{{download_dir}}/report.xlsx"
    - type: element_text_equals
      locator: "保存状态"
      value: "已保存"
```

第一版只要支持少量验证类型即可：

- 窗口标题包含。
- 页面文字包含。
- UIA 控件存在。
- 文件存在。
- 剪贴板内容匹配。

### 4.5 增加失败分级

建议把失败分成四类：

| 类型 | 示例 | 策略 |
| --- | --- | --- |
| 可重试 | 页面加载慢、按钮暂不可点 | 等待后重试 |
| 可降级 | UIA 找不到但 OCR 找得到 | 切换定位策略 |
| 需接管 | 登录、验证码、权限弹窗 | 暂停用户处理 |
| 终止 | 高风险动作不确定、状态无法验证 | 停止任务并输出报告 |

这会让错误恢复策略更工程化。

### 4.6 增加日志最小字段

执行日志建议明确最小字段：

```text
TaskRunLog
- task_id
- step_id
- timestamp
- action
- locator_strategy
- target
- before_screenshot
- after_screenshot
- ui_tree_snapshot
- expected_state
- verification_result
- error_type
- human_takeover_required
```

这对失败复盘、回归测试、用户信任都很重要。

## 5. 可以保留但要标记为后续

以下内容可以保留在路线图里，但建议在文档中显式标记为阶段 4 或阶段 5：

- 用户认证。
- 云端同步。
- OSS。
- ASR。
- 视频问答。
- 自动更新。
- 企业版管理后台。
- 团队记忆库。
- macOS / Linux。
- Python 脚本引擎。
- 本地多模态大模型。

## 6. 建议修改后的第一版边界

第一版建议只承诺：

- Windows。
- Electron 桌面客户端。
- 本地 Agent 服务。
- 截图和活动窗口识别。
- UIA 控件读取。
- Playwright 托管浏览器。
- 鼠标键盘执行。
- 手写流程记忆。
- 文字教学生成流程草稿。
- 流程编辑器。
- 本地 SQLite + 文件目录。
- 执行日志。
- 人工接管。
- 紧急停止。
- 3-5 条真实低风险流程模板。

第一版不承诺：

- 任意软件可操作。
- 云同步。
- 团队协作。
- 自动处理验证码。
- 自动支付、转账、删除、审批、群发。
- Python 脚本执行。
- macOS。
- 语音注释和视频问答。

## 7. 结论

新版方案已经具备可落地的骨架，但还需要进一步把“MVP 能力”和“商业版能力”分层。

最重要的修改是：

> MVP 架构只保留本地桌面 Agent、工具执行、流程记忆、流程编辑、日志和模型调用适配层。账号、云同步、OSS、ASR、视频问答、自动更新和企业能力全部移出主路径。

这样第一版才更可能在 6-10 周内跑出一个真正能演示、能试点、能迭代的桌面 Agent。
