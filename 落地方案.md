## 一、Agivar 产品分析

### 1.1 产品定位

**Agivar** 是一款运行在桌面端的 AI 助手，核心卖点：**「教一次、自动做」**。

用户通过录屏演示操作流程 → AI 学会后沉淀到记忆库 → 以后执行相关任务时自动复用。

### 1.2 两种核心模式

| | 任务模式（Task Mode） | 教学模式（Teach Mode） |
|---|---|---|
| **用途** | 让 AI 执行具体任务 | 教 AI 新的操作流程/规则/偏好 |
| **输入** | 自然语言指令 | 对话讲解 / 录屏演示 |
| **AI 行为** | 规划 TODO → 执行脚本/GUI操作 → 验证结果 | 理解讲解 → 整理成结构化知识 → 写入记忆库 |
| **产出** | 任务完成 + 总结 | 一条可复用的记忆条目 |

### 1.3 核心能力：录屏教学

1. **录制**：捕获鼠标/键盘/屏幕变化，支持文字注释（Alt+I）和语音注释（Alt+D）
2. **处理**：云端自动提取关键帧 → 逐帧标注 → 生成视频简介 → 生成结构化操作描述
3. **沉淀**：存入记忆库，按 `平台/主题`（如 `bilibili/投币`）归档
4. **复用**：任务模式执行前自动查阅记忆库，可就录屏细节追问

### 1.4 记忆库系统

- 云端存储，账号绑定，跨设备可用
- 按 `平台/主题` 组织（如 `公司报销系统/提交发票`、`general/工具调用偏好`）
- 两类内容：**操作流程类**（概述/初始状态/操作步骤/阶段性成果/相关视频）和**规则/偏好/概念类**
- AI 自动维护：先查索引再决定新增/更新，冲突以最新为准，不确定会反问

### 1.5 任务模式工作原理

- **先规划后执行**：列 TODO 计划，按计划推进
- **优先用"轻"的方式**：能用脚本/命令解决的不动 GUI
- **确定性任务写脚本**：生成 Python 脚本保存到 `~/.agivar/` 下
- **执行前查记忆库**：有相关经验就按照步骤执行
- **边做边验证**：每步操作前后观察状态确认效果
- **人工接管**：遇到密码/验证码/权限时暂停等待用户

### 1.6 界面组成

- 左侧：会话侧边栏（历史对话、筛选、删除）
- 中间：聊天区（消息、工具调用卡片、GUI 操作高亮）
- 底部：输入栏（文本、图片、录屏、教学模式开关）
- 浮窗：录制控制条、注释输入区、录屏预览、任务进度悬浮窗

### 1.7 系统要求

**第一版仅支持 Windows**（macOS 支持后置到阶段 5）：
- Windows 10 (1903+) / Windows 11
- 核心功能本地可用，无需登录即可使用
- 联网仅用于：多模态模型 API 调用、可选的软件更新检查
- 登录、云端同步、跨设备记忆库不是 MVP 必需能力（阶段 4+）
- 已沉淀的流程支持离线执行（本地记忆库 + 本地 embedding）
- 本地数据存放在 `~/.app-name/`

> 为什么第一版只做 Windows：桌面自动化在 Windows 上生态最完善（UI Automation、SendInput），企业目标用户以 Windows 为主，集中精力在一个平台做深做稳。macOS 支持在架构层预留（Rust Native Module 双平台抽象），但第一版不交付。

## 四、技术选型

| 层级 | 推荐方案 | 备选 | 选型理由 |
|---|---|---|---|
| **桌面框架** | Electron + React + TS | Tauri (Rust) | 生态成熟、跨平台一致性好、Native Module 支持 |
| **Native 语言** | Rust | C++ | 内存安全、跨平台编译好、Node Addon (napi-rs) |
| **录屏捕获** | DXGI 或 Windows Graphics Capture (阶段 0 对比选型) | libobs | 阶段 0 同时验证两者的稳定性、权限、窗口级捕获和性能 |
| **GUI 模拟** | SendInput (Win) / CGEvent (Mac) | PyAutoGUI | 原生精确控制 |
| **多模态 LLM** | Claude 4.x Vision API | GPT-4o / Qwen-VL | 视觉理解能力强、工具调用成熟 |
| **本地小模型** | Qwen2.5-VL-7B (端侧) | InternVL-2B | 简单场景降低云端调用，降成本 |
| **后端框架** | Go (Fiber/Gin) | Node.js (Fastify) | 高并发、部署简单 |
| **数据库** | SQLite (本地) + PostgreSQL (云端同步) | MongoDB | 本地优先保证离线可用，云端同步跨设备 |
| **向量检索** | sqlite-vec (本地) + pgvector (云端) | Milvus / Qdrant | 本地检索零延迟，云端备份 |
| **对象存储** | 本地文件目录 + 阿里云 OSS (同步) | MinIO (自建) | 关键帧/录屏本地存储，按需同步云端 |
| **任务队列** | Redis + Asynq (Go) | RabbitMQ | 录屏处理异步流水线 |
| **ASR** | FunASR (阿里开源) | Whisper / 讯飞 | 中文效果好、可自部署 |
| **自动更新** | electron-updater | Sparkle (Mac) + Squirrel (Win) | Electron 生态标配 |
| **文档站** | Docusaurus | VitePress | React 生态 |
| **状态管理** | Zustand | Jotai | 轻量、TypeScript 友好 |

### 4.1 开源库整合策略

本产品不从零手写桌面自动化底层能力。第一版采用“成熟开源库打底 + 自研产品闭环”的方式推进：底层能力优先复用社区验证过的库，上层围绕教学、流程记忆、安全控制、执行日志和失败恢复做产品化封装。

#### Phase 0 必须验证的开源库

| 能力 | 开源库 / 工具 | 用途 | Phase 0 验证重点 |
|---|---|---|---|
| 桌面客户端 | Electron + React + TypeScript | 桌面壳、验证面板、IPC 通信 | Electron 主进程可加载 native addon，packaged 环境路径正确 |
| Monorepo 构建 | pnpm + Turborepo + Vite / electron-vite | 多包管理、构建和开发体验 | native → core → desktop 构建链路稳定 |
| Native 绑定 | napi-rs | 将 Rust 能力暴露给 Node.js / Electron | Node.js、Electron、packaged 三种环境加载成功 |
| 键鼠控制 | `@nut-tree/nut-js` | 鼠标点击、键盘输入、快捷键、滚动 | 记事本输入成功率、焦点窗口校验、DPI 下点击偏差 |
| 截屏 | `node-screenshots` | 主屏截图、窗口截图能力验证 | 主屏截图成功率、活动窗口识别、窗口截图边界 |
| 浏览器自动化 | Playwright | Chrome / Edge 表单填写、DOM 定位 | 托管浏览器、本地测试页、表单提交成功率 |

| 录屏捕获 | `windows-capture` + `win_desktop_duplication` | 分别验证 WGC 和 DXGI 路线 | 帧率、丢帧率、CPU 占用、窗口级捕获、资源释放 |

#### 自研边界

开源库只解决底层能力，以下部分必须自研：

- **统一工具接口**：把 Playwright、UIA、视觉定位、坐标点击封装为一致的 Agent 工具。
- **安全控制层**：风险分级、二次确认、紧急停止、人工接管、敏感信息保护。
- **流程记忆模型**：`WorkflowMemory`、`WorkflowStep`、`TaskRun`、`TaskStepLog` 等核心数据结构。
- **执行闭环**：观察 → 执行 → 验证 → 失败分类 → 重试 / 降级 / 人工接管。
- **教学模式**：把文字教学或录屏内容转换成可编辑、可复用、可验证的流程。
- **执行日志与回放**：每步截图、UIA 快照、动作参数、验证结果和错误原因。
- **产品体验**：聊天界面、流程编辑器、录制控制条、任务进度浮窗、权限隐私引导。

## 六、关键实现要点

### 6.1 录屏引擎（核心技术难点）

**设计原则**：
- 采用**事件驱动**的关键帧提取（不是固定帧率录制）
- 每次鼠标点击/按键前截一帧 + 操作后截一帧
- 安静时段每 2-3 秒定时采样
- 浮窗排除：获取窗口句柄/位置，截图时跳过这些区域

**已知技术难点及应对**：

| 难点 | 平台 | 应对方案 |
|---|---|---|
| HDR 显示器色彩空间 | Windows | DXGI 输出 → 统一转换到 sRGB 后保存 |
| 多屏异构 DPI | Windows | 每个屏幕独立获取 DPI 缩放，统一到逻辑坐标系 |
| 浮窗排除闪烁/残影 | macOS | ScreenCaptureKit 的 `excludingWindows` 参数 + 维护窗口 ID 列表 |
| Electron 自身 GPU 占用影响截屏 | 全平台 | 录屏引擎在独立线程运行，避免与渲染进程争抢 GPU |
| 注释输入期间的事件过滤 | 全平台 | 注释框获焦期间标记 `annotation_active`，过滤键盘事件 |

**关键帧提取策略**：
```
触发关键帧的事件：
├── 鼠标左键/右键按下（截操作前的画面）
├── 鼠标左键/右键释放后 300ms（截操作后的画面）
├── 鼠标拖拽开始/结束
├── 键盘 Enter/Tab/Escape 按下
├── 快捷键组合（Ctrl+C, Ctrl+V 等）
├── 鼠标滚轮（累积 > 500px 距离后）
└── 定时器（无操作超过 3 秒时采样一帧）

过滤规则：
├── annotation_active 期间的键盘事件 → 丢弃
├── 连续两帧间隔 < 200ms → 合并（取后者）
└── 注释快捷键本身的 keydown/keyup → 丢弃
```

**数据格式**：
```json
{
  "recording_id": "rec_001",
  "duration_ms": 45000,
  "screen_info": {
    "width": 1920, "height": 1080,
    "dpi_scale": 1.25,
    "monitor_id": "primary"
  },
  "frames": [
    {"ts": 0, "type": "initial", "file": "frame_000.png"},
    {"ts": 3200, "type": "before_click", "file": "frame_001.png"},
    {"ts": 3500, "type": "after_click", "file": "frame_002.png"}
  ],
  "events": [
    {"ts": 3200, "type": "mouse_click", "x": 450, "y": 320, "button": "left",
     "physical_x": 562, "physical_y": 400},
    {"ts": 5100, "type": "key_input", "text": "bilibili.com"},
    {"ts": 5800, "type": "key_press", "key": "Enter"}
  ],
  "annotations": [
    {"ts": 8000, "type": "text", "content": "等页面加载完再点"},
    {"ts": 15000, "type": "voice", "content": "这里要选第一个搜索结果"}
  ]
}
```

### 6.2 操作描述生成（核心 AI 能力）

**输入**：关键帧图片序列 + 事件时间轴 + 用户注释

**Prompt 设计要点**：
- 约束输出格式：`概述 / 初始状态 / 操作步骤（含阶段性成果） / 相关视频`
- 界面元素描述要**消歧义**（"点击右下角蓝色'确认'按钮"而非"点确认"）
- 每个抽象步骤结尾有**阶段性成果**（用于验证执行是否成功）
- 复用用户注释中的术语和分段方式

**输出示例**：
```markdown
# 概述
演示在 B 站搜索并给视频投币的操作流程。

# 初始状态
浏览器已打开，当前在任意页面。

# 操作步骤

## 第一步：打开 B 站
1. 点击浏览器地址栏
2. 输入 bilibili.com
3. 按 Enter
阶段性成果：进入 B 站首页，可见搜索框和推荐视频。

## 第二步：搜索目标
1. 点击顶部搜索框
2. 输入"非十科技"
3. 点击搜索框右侧放大镜图标
阶段性成果：搜索结果页显示，列表中有多个视频。

## 第三步：投币
1. 点击第一个视频进入播放页
2. 点击视频下方"投币"按钮（硬币图标）
3. 在弹窗中点击"+2"选择投 2 个币
4. 点击弹窗右下角蓝色"确定"按钮
阶段性成果：投币成功提示出现，投币按钮状态变为已投。

# 相关示教视频
视频 ID: rec_001; 简介: B 站投币操作演示
```

### 6.3 Computer Use 执行引擎

**定位优先级（从高到低）**：

桌面 Agent 不应一上来就用截图点坐标。按以下优先级选择定位策略：

| 优先级 | 定位方式 | 适用场景 | 说明 |
|---|---|---|---|
| 1 | **浏览器 DOM 定位** | 目标在浏览器中 | 通过 Playwright 使用 role、label、text、selector 精确定位 |
| 2 | **Windows UI Automation** | 原生桌面应用 | 通过控件树定位按钮、文本框、菜单、列表 |
| 3 | **OCR + 视觉定位** | 控件树不可用时 | 截图 + OCR + 多模态模型识别目标区域 |
| 4 | **坐标兜底** | 以上均失败时 | 仅作为当前会话兜底，不写入长期流程记忆 |
| 5 | **人工接管** | 信心不足或高风险 | 暂停并让用户处理 |

> 关键原则：坐标只能作为当前会话的临时手段，不作为长期流程记忆的主路径。录屏教学沉淀的应该是"语义流程"而非坐标回放。

**浏览器会话策略**：

桌面 Agent 做浏览器任务时，是否使用用户已有浏览器会话很关键。

| 模式 | 说明 | 适用场景 |
|---|---|---|
| **独立托管浏览器**（默认） | Playwright 启动独立 Chromium 实例，与用户主浏览器完全隔离 | 不需要登录态的公开网页操作 |
| **托管浏览器内登录** | 用户在 Playwright 托管的浏览器中手动登录，系统复用该会话 | 需要登录态但要求隔离的场景 |

> 第一版不直接控制用户主浏览器（Chrome / Edge），除非有明确授权和隔离策略。原因：避免污染用户日常浏览器、干扰插件/扩展、破坏 SSO 会话。
>
> 后续可评估通过 CDP 连接用户浏览器的方案，但需要解决：用户数据目录隔离、插件兼容性、企业 SSO/证书/代理处理。

**执行循环**：
```
1. 接收任务 → 查记忆库有无相关操作描述
2. 如有匹配的记忆 → 按步骤执行：
   a. 检查当前画面是否符合"初始状态"
   b. 按操作步骤逐步执行
   c. 每步执行后截屏，与"阶段性成果"对照验证
   d. 验证失败 → 重试 1 次 → 仍失败 → 切到 LLM 自主规划
3. 如无匹配记忆 → LLM 自主规划：
   a. 截屏 → 发给 LLM 理解当前画面
   b. LLM 决定下一步动作
   c. 执行动作 → 等待 1-2 秒 → 截屏验证效果
   d. 循环直到完成或需要人工接管
```

**延迟处理策略**：
截屏→上传→LLM推理→返回结果的 round trip 约 2-5 秒，应对方案：
- GUI 操作时在屏幕上显示"AI 正在思考..."的高亮提示
- 简单操作（如文字输入）本地直接执行，不等待 LLM 确认
- 批量操作时，LLM 一次返回多步指令，客户端顺序执行
- 未来引入本地小模型做简单判断（如"页面是否加载完"）

**人工接管触发条件**：
- 检测到密码输入框
- 检测到验证码/CAPTCHA
- 需要系统权限弹窗
- LLM 连续 3 次操作未产生预期变化
- 用户按下紧急停止热键（默认 Ctrl+Alt+Space，可自定义）

**风险动作四级分级**：

| 风险等级 | 示例 | 执行策略 |
|---|---|---|
| **低风险** | 查询、筛选、复制、导出、打开文件 | 可自动执行，仅记录日志 |
| **中风险** | 新增记录、修改字段、上传附件 | 执行前展示计划，必要时弹窗确认 |
| **高风险** | 删除、提交审批、批量发送、覆盖文件 | 强制人工确认后才执行 |
| **禁止** | 代输验证码、绕过 2FA、自动支付/转账 | 必须人工接管，系统不执行 |

**操作安全基础设施**：
- **文件操作快照**：修改/删除文件前自动备份到 `~/.app-name/undo/`
  - 注意：仅文件操作可快照。网页提交、第三方系统写入、消息发送、支付等操作无法撤销
  - 不对用户承诺"通用 Undo"，而是通过执行前确认 + 高风险暂停 + 可回放日志来保障安全
- **执行日志**：每步操作记录截图+描述+时间戳，可在任务进度浮窗查看
- **紧急停止**：全局热键（默认 Ctrl+Alt+Space，可自定义）+ 托盘图标按钮 + 悬浮窗停止按钮，立即中止所有操作。不使用 Ctrl+Shift+Esc（与 Windows 任务管理器冲突）
- **每步必须验证**：每个步骤都要有 `expected_state`（窗口标题、页面文字、按钮状态等），没有验证条件的步骤不允许作为稳定自动化步骤

**状态验证 DSL**：

`expected_state` 使用 YAML 格式定义，第一版支持以下验证类型：

```yaml
expected_state:
  any:                              # 满足任一条件即通过
    - type: window_title_contains
      value: "客户管理"
    - type: page_text_contains
      value: "保存成功"
    - type: file_exists
      path: "{{download_dir}}/report.xlsx"
    - type: element_text_equals
      locator: "保存状态"
      value: "已保存"
    - type: uia_element_exists
      name: "提交按钮"
      control_type: "Button"
    - type: clipboard_contains
      value: "已复制的内容"
```

第一版验证类型清单：

| 类型 | 说明 | 示例 |
|---|---|---|
| `window_title_contains` | 窗口标题包含指定文字 | 打开了目标页面 |
| `page_text_contains` | 页面/窗口内出现指定文字 | "保存成功"提示 |
| `uia_element_exists` | UIA 控件存在 | 某个按钮出现 |
| `file_exists` | 文件存在 | 下载完成 |
| `element_text_equals` | 元素文本等于指定值 | 状态字段变为"已提交" |
| `clipboard_contains` | 剪贴板内容包含 | 复制操作成功 |

> 验证条件支持 `any`（满足任一）和 `all`（全部满足）两种组合。后续可扩展更多类型。

**失败分级策略**：

步骤执行失败不应一刀切处理，按严重程度分四类：

| 类型 | 示例 | 策略 |
|---|---|---|
| **可重试** | 页面加载慢、按钮暂不可点击、网络超时 | 等待后自动重试（最多 2 次） |
| **可降级** | UIA 定位失败但 OCR 可识别 | 切换到下一优先级定位策略 |
| **需接管** | 登录页、验证码、权限弹窗、未知对话框 | 暂停任务，用户处理后继续 |
| **终止** | 高风险动作不确定、状态完全无法验证、连续失败 | 停止任务并输出诊断报告 |

> 失败类型由系统自动判断，也可在流程步骤的 `fallback` 字段中显式指定。

### 6.4 记忆库检索

**检索流程**（本地优先）：
1. 任务描述 → 本地 embedding 模型生成向量
2. 在本地 sqlite-vec 中做语义相似度搜索
3. 返回 Top-K 相关条目（阈值过滤）
4. 将匹配的记忆注入 Agent 的 context
5. （离线模式下全流程可用，无需云端）

**索引设计**：
- 本地 SQLite 存储完整记忆索引
- 启动时加载索引摘要给 Agent（每条一行：`平台/主题 + 简介`）
- Agent 可根据索引决定是否需要读取某条记忆的全文

**版本管理**：
- 每次更新记忆条目时保留旧版本（记录更新时间 + 更新原因）
- 支持在对话中让 AI 回滚到历史版本
- 冲突策略：最新明确说明 > 旧版本

**云端同步**：
- 用户登录后，本地记忆增量同步到云端
- 新设备登录时，从云端拉取全量记忆到本地
- 冲突解决：时间戳最新者优先，冲突记录保留供用户查看

### 6.5 脚本执行（后置到阶段 5）

> 第一版不开放 Python 脚本执行。桌面 Agent 已具备较高系统权限，再允许模型生成脚本会显著增加安全风险。

**第一版只允许固定工具调用**：
- 文件读取/复制/移动
- CSV/Excel 数据读取
- 浏览器 Playwright 操作
- UIA 控件操作
- 鼠标键盘动作
- 截屏和状态观察

**Python 脚本引擎上线条件**（阶段 5）：
- 用户显式批准每次执行
- 运行前展示完整代码
- 限制工作目录（只能访问用户指定目录 + `~/.app-name/scripts/`）
- 默认无网络（禁止 `socket`、`requests` 除非白名单）
- 默认无系统命令（禁止 `subprocess`、`os.system`、`ctypes`）
- 使用 `RestrictedPython` 或自定义 import hook
- 执行超时：默认 60 秒

### 6.6 统一工具接口抽象层

上层 Agent 不应直接关心底层是 UIA、Playwright、OCR 还是坐标点击。所有桌面能力封装为统一工具接口：

```
Desktop Tool Interface
├── observe_screen()              // 截取当前主屏幕
├── observe_active_window()       // 截取活动窗口
├── get_ui_tree(window_id)        // 获取窗口控件树 (UIA)
├── click(target)                 // 点击目标 (自动选择定位策略)
├── double_click(target)          // 双击
├── right_click(target)           // 右键点击
├── type_text(text)               // 输入文本
├── press_hotkey(keys)            // 按快捷键
├── scroll(direction, amount)     // 滚动
├── drag(from, to)                // 拖拽
├── wait_for(condition)           // 等待条件满足
├── ask_user_takeover(reason)     // 请求人工接管
└── verify_state(expected)        // 验证当前状态
```

**target 定位自动路由**：当 Agent 调用 `click(target)` 时，系统内部按优先级自动选择：
1. 如果目标在浏览器中 → Playwright DOM 定位
2. 如果窗口有 UIA 控件树 → UI Automation 定位
3. 如果以上不可用 → OCR + 视觉模型定位
4. 最后兜底 → 坐标点击（仅当前会话有效）

### 6.7 结构化数据模型

流程记忆和任务执行使用以下核心数据模型：

```
WorkflowMemory（流程记忆）
├── id                     // 唯一标识
├── app_name               // 应用名称 (如 "Chrome", "企业微信")
├── platform               // desktop | browser | hybrid
├── topic                  // 主题 (如 "bilibili/投币")
├── trigger_examples[]     // 触发示例 (如 "帮我给这个视频投币")
├── summary                // 流程概述
├── initial_state          // 初始状态描述
├── steps[]                // → WorkflowStep
├── success_criteria       // 整体成功判断
├── risk_level             // low | medium | high
├── source_type            // text | recording | edited
├── source_recording_id    // 关联录屏 ID
├── version                // 版本号
├── created_at / updated_at

WorkflowStep（流程步骤）
├── id
├── memory_id              // 所属流程
├── order                  // 步骤顺序
├── intent                 // 步骤意图 (如 "点击搜索框")
├── target_hint            // 目标描述 (如 "顶部搜索框，placeholder 为'搜索'")
├── locator_strategy       // 定位策略: playwright | uia | vision | coordinate | human
├── target_locator         // 具体定位器 (如 selector、UIA path、区域描述)
├── input_hint             // 输入内容 (如 "非十科技")
├── expected_state         // 步骤完成后的验证条件
├── fallback               // 失败时的处理策略
├── risk_level             // 该步骤的风险等级

Recording（录屏记录）
├── id
├── app_name / window_title
├── raw_video_path         // 原始视频
├── event_log_path         // 事件时间轴
├── keyframes[]            // 关键帧列表
├── annotations[]          // 用户注释
├── generated_workflow_id  // 生成的流程记忆 ID
├── created_at

TaskRun（任务执行记录）
├── id
├── user_goal              // 用户目标
├── selected_memory_ids[]  // 匹配的记忆
├── plan                   // 执行计划
├── status                 // pending | running | paused | success | failed
├── step_logs[]            // → TaskStepLog
├── screenshots[]          // 过程截图
├── ui_tree_snapshots[]    // UIA 快照
├── human_takeover_events[]// 人工接管事件
├── started_at / finished_at

TaskStepLog（步骤级执行日志）
├── task_id                // 所属任务
├── step_id                // 对应流程步骤
├── timestamp              // 执行时间
├── action                 // 执行的动作 (click/type/scroll/hotkey...)
├── locator_strategy       // 实际使用的定位策略
├── target                 // 实际定位目标
├── before_screenshot      // 执行前截图路径
├── after_screenshot       // 执行后截图路径
├── ui_tree_snapshot       // 执行时 UIA 快照路径
├── expected_state         // 验证条件
├── verification_result    // pass | fail | skipped
├── error_type             // null | retryable | degradable | takeover | terminal
├── human_takeover_required// 是否触发人工接管
```

> 关键设计：每个 `WorkflowStep` 都有独立的 `locator_strategy`、`expected_state` 和 `fallback`，使流程在不同环境下具备自适应能力。`TaskStepLog` 为每步操作提供完整审计链——失败复盘、回归测试、用户信任都依赖这些字段。

## 十、风险与应对

### 10.1 MVP 核心风险

> 第一版 Windows MVP 最可能遇到的风险，必须在阶段 0-2 期间重点关注。

| 风险 | 影响 | 概率 | 应对措施 |
|---|---|---|---|
| **UIA 控件读取不稳定** | 不同软件控件结构差异大，定位失败 | 高 | ① 第一版限定 2-3 个目标应用 ② UIA → 视觉 → 人工的降级链 ③ 评测集回归 |
| **截图和 DPI 映射错误** | 多缩放率下坐标偏移，点击偏离 | 高 | ① 阶段 0 验证 100%/125%/150% ② 统一使用逻辑坐标 ③ 多 DPI CI 测试 |
| **LLM 视觉理解不准确** | 操作描述质量差，执行出错 | 高 | ① 多模型 A/B 测试选型 ② 人工校正 workflow ③ 用户可编辑操作描述 |
| **Computer Use 成功率低** | 用户体验差 | 高 | ① 有记忆时按步骤执行 ② 无记忆时分级协作 ③ 优先稳定定位 |
| **LLM API 成本过高** | 亏损 | 高 | ① 缓存+批量调用 ② 分级处理 ③ 本地小模型 ④ 定价覆盖成本 |
| **Playwright 与用户浏览器会话打通困难** | 无法复用登录态 | 中 | ① 默认独立托管浏览器 ② 用户在托管浏览器中登录 ③ 不控制主浏览器 |
| **高风险动作误执行** | 误发消息/误提交/误删除 | 中 | ① 四级风险分级 ② 高风险强制确认 ③ 禁止类动作必须人工 |
| **任务失败无法诊断** | 无法改进 | 中 | ① 每步截图+日志 ② 结构化失败分类 ③ UI 树快照 |
| **杀毒/安全软件拦截** | 键鼠模拟被拦截 | 中 | ① 代码签名 ② 引导加白名单 ③ 常见杀软兼容性测试 |
| **用户不信任桌面权限** | 采纳率低 | 中 | ① 首次启动权限隐私引导 ② 渐进式授权 ③ 操作预览 ④ 详细日志 ⑤ 一键停止 |

### 10.2 后续阶段风险

> 这些风险对第一版 Windows MVP 不是核心问题，但需要在路线图中持续跟踪。

| 风险 | 影响 | 阶段 | 应对措施 |
|---|---|---|---|
| **macOS 权限弹窗多** | 功能受限 | 阶段 5 | ① 引导式权限申请 ② 最小权限原则 |
| **跨平台一致性** | Win/Mac 行为差异 | 阶段 5 | ① Rust Native Module 双平台实现 ② 跨平台 CI |
| **云端隐私合规（个保法/GDPR）** | 法律风险 | 阶段 4 | ① 隐私政策告知 ② 可选不上传 ③ 数据删除 API |

## 十二、关键决策记录

| 决策 | 选择 | 理由 |
|---|---|---|
| 桌面框架 | Electron 而非 Tauri | 生态成熟度、Native Module 支持、跨平台一致性 |
| Native 语言 | Rust 而非 C++ | 内存安全、napi-rs 生态、跨平台编译 |
| MVP 阶段顺序 | 先执行后教学 | 执行闭环不稳定，教学再好也无法转化价值 |
| 第一版平台 | 仅 Windows，macOS 后置 | Windows 生态最完善（UIA/SendInput），企业用户以 Windows 为主 |
| 记忆库架构 | 本地优先 + 云端同步 | 离线可用降低依赖，登录后同步保证跨设备 |
| 定位策略 | Playwright → UIA → 视觉 → 坐标 → 人工 | 稳定性递减，坐标不入长期记忆 |
| 风险分级 | 四级：低/中/高/禁止 | 明确哪些可自动、哪些要确认、哪些必须人工 |
| Computer Use 策略 | 有记忆按步骤执行 > 无记忆 LLM 规划 > 脚本 | 有记忆时成功率远高于纯 LLM 自主操作 |
| 定价 | ¥99/月起步 | 需覆盖 LLM API 成本（约 ¥120-180/用户/月降本后） |
| 第一版脚本执行 | 不开放，只允许固定工具调用 | 桌面 Agent 已有高权限，脚本执行增加安全风险，需更严格管控 |
| 紧急停止热键 | Ctrl+Alt+Space（可自定义） | 避免与 Windows 任务管理器 Ctrl+Shift+Esc 冲突 |
| 团队规模 | 11 人 | 录屏引擎和 GUI 操控各需专人，AI Prompt 迭代频繁需独立投入 |
